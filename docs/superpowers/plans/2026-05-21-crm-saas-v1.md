# CRM SaaS v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the first working version of a multi-workspace CRM SaaS for architecture-agency operations, including roles, assistant chat, CRM records, outreach, project operations, templates, documents, content planning, and feedback inboxes.

**Architecture:** Use a TypeScript monorepo with a Next.js web app, PostgreSQL database, object storage for files, and service modules with clear domain boundaries. CRM data remains the source of truth; Google integrations are optional sync/export layers.

**Tech Stack:** Next.js, React, TypeScript, PostgreSQL, Supabase Auth/Storage or equivalent, Prisma, Tailwind CSS, shadcn/ui, Radix UI, TanStack Query, TanStack Table, background jobs through BullMQ/Redis when needed.

---

## Scope Split

This is a large v1 and should be implemented as connected milestones, not one long unreviewed change. Each milestone must produce running software.

Milestone order:

1. Foundation and workspace auth.
2. Core CRM schema and CRUD.
3. Assistant shell and feedback capture.
4. Documents, templates, and attachments.
5. L01 lead intake and KP generation.
6. L02 outreach and CSV import.
7. L03 project operations.
8. L04 content factory.
9. Google integration hooks.
10. Platform admin route.

Responsibilities:

- `apps/web`: Next.js routes, layouts, server actions/API adapters, UI composition.
- `packages/db`: Prisma schema, migrations, typed database client.
- `packages/auth`: Google login, workspace membership, permissions.
- `packages/core`: domain services for clients, leads, projects, outreach, content, feedback.
- `packages/assistant`: assistant thread/message/action model, intent classification, action preview.
- `packages/documents`: DOCX template parsing, generation, PDF export orchestration.
- `packages/integrations`: Google Drive/Calendar/Gmail adapters, PromoRepublic/Hubstaff placeholders.
- `packages/ui`: reusable themed UI primitives and design tokens.

- [ ] **Step 2: Implement deterministic v1 classifier**

Create `packages/assistant/src/classify-intent.ts`:

```ts
export type AssistantIntent =
  | "crm_action"
  | "support_request"
  | "bug_report"
  | "feature_request"
  | "ux_feedback"
  | "business_process_note"
  | "permission_blocked"
  | "other";

export function classifyIntent(message: string): AssistantIntent {
  const text = message.toLowerCase();

  if (/(create|add|generate|schedule|update|создай|добавь|сгенерируй|поставь)/.test(text)) {
    return "crm_action";
  }

  if (/(bug|broken|does not work|не работает|ошибка)/.test(text)) {
    return "bug_report";
  }

  if (/(feature|please add|would be nice|добавить|хотелось бы)/.test(text)) {
```

```ts
export const AVAILABLE_DOCUMENT_VARIABLES = [
  "date",
  "client_name",
  "client_address_line_1",
  "client_address_line_2",
  "project_name",
  "project_address",

  "wohnflaeche",

  "total_net",
  "mwst",
  "total_gross",
  "ms1_net",
  "ms2_net",
  "ms3_net",
  "offer_valid_until"
] as const;

export function parsePlaceholders(text: string): string[] {
  const matches = text.matchAll(/\{\{\s*([^{}]+?)\s*\}\}/g);
  return Array.from(new Set(Array.from(matches, (match) => match[1].trim()))).sort();
}

export function findUnknownPlaceholders(placeholders: string[], allowed: readonly string[]): string[] {
  const allowedSet = new Set(allowed);
  return placeholders.filter((placeholder) => !allowedSet.has(placeholder)).sort();
}
```
